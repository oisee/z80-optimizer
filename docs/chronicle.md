## Глава 1: День рождения и 21 миллиард кандидатов

Утро 26 марта 2026 года. День рождения. На столе — торт, два монитора,
и кластер из четырёх машин, каждая со своим характером:

- **i7 (main)**: 2× RTX 4060 Ti 16GB — рабочие лошадки, CUDA 12.0
- **i5 (ubullama)**: RTX 2070 8GB — верификатор, второй фронт
- **i3 (macaron)**: Radeon RX 580 8GB — AMD-доказательство, Mesa OpenCL
- **M2 MacBook Air**: Metal — яблочное доказательство, кофейный столик

Идея простая: Z80 — процессор 1976 года. Состояние — 11 байт: семь регистров
(A, B, C, D, E, H, L), флаги F, указатель стека SP и виртуальный байт памяти M.
Одиннадцать байт — это ничто для GPU. Каждая инструкция детерминированно
трансформирует эти 11 байт. Две последовательности инструкций эквивалентны тогда
и только тогда, когда они дают одинаковый результат для ВСЕХ возможных входов.

Всего возможных входов — примерно 2^88. Перебирать все — нереально.
Но есть трюк, который делает задачу решаемой: тест-векторы.

### Three-Stage Pipeline

**QuickCheck**: 8 тщательно подобранных входов. Отбрасывает 99.99% кандидатов
за микросекунды. Если две последовательности не согласны хотя бы на одном из 8 —
они точно не эквивалентны.

**MidCheck**: ещё 24 вектора, нацеленных на BIT/SET/RES — инструкции, чьё
поведение зависит от конкретной позиции бита. Ловит ещё 23% false positives.

**ExhaustiveCheck**: для ~0.01% выживших — полный перебор всех возможных входов.
На GPU: 256 потоков на кандидата, поток *i* обрабатывает A=*i* и перебирает
остальные регистры. Shared-memory early termination — как только один поток нашёл
различие, все останавливаются.

Запускаем на двух RTX 4060 Ti. Pipeline обрабатывает 17.8 миллионов пар
инструкций за шесть минут.

Результат: **739,575 доказанных правил peephole-оптимизации.**

Каждое — математическая теорема: "эти две последовательности инструкций дают
идентичный выход для всех возможных входов."

Некоторые — очевидные:

```asm
LD A, B : LD B, A  →  LD A, B    ; вторая инструкция избыточна
```

Другие — глубоко неожиданные:

```asm
SLA A : RR A  →  OR A    ; экономия 12 T-states, 3 байта
```

Shift left, потом rotate right — равно OR с самим собой? Это работает, потому что
shift очищает бит 0, rotate вставляет старый бит 7 через carry, и результирующий
эффект на флаги совпадает с OR A. Ни один человек не спроектировал бы это правило.
GPU не нужно понимать *почему* — он просто доказывает *что*.

739 тысяч правил. Шесть минут. Торт ещё даже не порезали.

## Глава 2: Когда 3 инструкции хватает на всё

У Z80 нет инструкции умножения. Чтобы умножить A на константу K, нужна
последовательность сдвигов, сложений и вычитаний. Какая последовательность
самая короткая для каждого K от 2 до 255?

### 8-битное умножение: 14 из 21

Начали с 21 инструкции-кандидата: сдвиги, ротации, ADD, SUB, NEG, carry.
GPU нашёл оптимальные последовательности для 103 констант при длине <= 8.

Потом мы посмотрели, какие инструкции *реально появляются* в оптимальных
решениях. Семь — ни разу:

- `SLA A` — то же самое, что `ADD A,A`, но 8T вместо 4T (строго хуже)
- `RLC A`, `RRC A` — CB-prefix версии RLCA/RRCA (8T vs 4T)
- `OR A`, `SCF`, `EX AF,AF'` — теоретически полезны, но никогда оптимальны

Убираем их. Pool сжимается с 21 до 14 инструкций. При длине 9 это
38× меньше кандидатов. При длине 10 — 57×. При длине 11 — 86×.

**Empirical pool reduction** — самая мощная техника всего проекта.

С 14 инструкциями GPU нашёл:
- 164 константы при длине <= 9
- 249 констант при длине <= 10
- последние 5 (170, 171, 173, 179, 181) при длине 11

**Все 254 нетривиальные константы имеют доказанно оптимальные последовательности.**

Самое красивое решение: `×255 = NEG` — одна инструкция, 8 T-states. Потому что
255 = -1 mod 256, и NEG вычисляет -A. GPU не знает теорию чисел. Он просто
обнаружил, что NEG даёт правильный результат для всех 256 входов.

```asm
; ×255: A = -A = A × 255 (mod 256)
NEG           ; 1 инструкция, 8T. Потому что -1 ≡ 255.
```

### 16-битное умножение: чудо трёх инструкций

Для 16-bit multiply (A × K → HL) история ещё драматичнее. Начали с 23
инструкций. GPU нашёл ВСЕ 254 константы... используя только **три**:

```asm
ADD HL,HL     ; удвоить 16-битный аккумулятор     (11T)
ADD HL,BC     ; прибавить оригинальный вход        (11T)
LD C,A        ; сохранить вход в C (BC = input)    (4T)
```

Всё. Каждое 16-битное умножение на Z80 — это последовательность doubles и adds.
Пространство поиска схлопнулось с 23^N до 3^N — в **13,600 раз при длине 8**.

Вся таблица — 30 секунд.

Потом добавили `SWAP_HL` (LD H,L / LD L,0 — это умножение на 256) и
`SUB HL,BC`. Они улучшили 88 из 254 констант (35%):

```asm
; ×255 (16-bit): 256 - 1 = byte swap, потом вычесть
SWAP_HL       ; HL = input × 256
LD C,A        ; save input
SUB HL,BC     ; HL = input×256 - input = input×255
; 3 виртуальные операции, 5 реальных Z80 инструкций, 30 T-states
```

Таблица частот говорит сама за себя:

| Instruction | Count | Share  |
|-------------|------:|-------:|
| ADD HL,HL   | 1282  | 54.0%  |
| ADD HL,BC   |  712  | 30.0%  |
| LD C,A      |  247  | 10.4%  |
| SUB HL,BC   |  127  |  5.3%  |
| SWAP_HL     |    7  |  0.3%  |

Одна инструкция — ADD HL,HL — это больше половины всего кода во всех 254
решениях. Удвоение правит миром.

*"GPU не угадывает. Он знает."*

## Глава 3: Деление — это умножение наоборот

Z80 не умеет делить. Деление на 10 (критично для вывода десятичных чисел)
обычно использует цикл на ~120 T-states. Можно ли лучше?

### Reciprocal trick

Классическая идея из Hacker's Delight: n/K = (n × M) >> S, где M ≈ 2^S/K.
Деление превращается в умножение на "магическую константу" и правый сдвиг.

Что нового: мы автоматизировали весь процесс.

**Шаг 1. Abstract chain solver** (CPU, 8 секунд на все 254 константы) находит
кратчайший паттерн multiply-then-shift. Эти цепочки ISA-independent — они
описывают *что* вычислять, а не *как*.

**Шаг 2. Focused GPU kernel** (только 6 инструкций вместо 37!) ищет конкретную
Z80-реализацию этой цепочки.

6 инструкций — это всё, что нужно для division:

| Instruction | Role               |
|-------------|---------------------|
| ADD HL,BC   | Reciprocal accumulate |
| ADD HL,HL   | Shift left          |
| LD A,H      | Extract quotient    |
| LD C,A      | Save multiplicand   |
| SHR_HL      | Shift right         |
| SRL A       | Logical shift A     |

### Guided brute-force

Ключевая архитектурная идея: **разделить математическую структуру и
ISA-специфичную реализацию.** Abstract chain говорит: "умножь на 171, сдвинь
вправо на 9." GPU находит: "какие регистры, в каком порядке."

Чистый brute-force с 37 инструкциями потребовал бы 37^14 ≈ 10^22 evaluations —
годы на любом GPU. Guided search с 6 инструкциями: 6^16 ≈ 10^12 — **одиннадцать
секунд**.

### Результаты

**Все 247 делителей (не степени двойки) — COMPLETE.**

```
/3   = 14 инструкций, 130T     (reciprocal ×171 + SRL)
/5   = 14 инструкций, 127T     (reciprocal ×205 + SRL)
/10  = 14 инструкций, 124T     (Hacker's Delight, найдено автоматически!)
/19  = 10 инструкций, 86T
/57  = 6 инструкций, 60T       (самый быстрый!)
/100 = 9 инструкций, 105T
/171 = 4 инструкции, 27T       (рекордсмен по краткости)
```

div10 = 124T — **в точности совпадает** с ручной оптимизацией из Hacker's Delight.
Но GPU нашёл это автоматически. За 11 секунд. На день рождения.

## Глава 4: Скала невозможности на 6 переменных

Для каждой теоретически возможной комбинации виртуальных регистров, их ширин,
ограничений и графов интерференции — до 6 переменных — мы вычислили доказанно
оптимальное физическое назначение. Или доказали, что его не существует.

**83.6 миллиона исчерпывающих записей.**

Результаты обнаруживают dramatic phase transition:

```
2 vregs:  95.9% feasible    — почти всё работает
3 vregs:  88.5% feasible    — ещё нормально
4 vregs:  78.7% feasible    — начинает поджимать
5 vregs:  67.7% feasible    — треть невозможна
6 vregs:   0.9% feasible    ← СКАЛА
```

При 6 виртуальных регистрах **99.1% всех возможных конфигураций не имеют
допустимого назначения на Z80.** Регистровый файл "заполняется" — физически
не хватает регистров для большинства комбинаций.

### 97.7% для реальных программ

Мы протестировали 129 уникальных сигнатур функций из реального компиляторного
корпуса (8 языков программирования, 1,567 функций). Для функций с 7-15
виртуальными регистрами:

- **3 feasible** — доказанно оптимальные назначения найдены
- **126 infeasible** — доказано, что допустимого назначения НЕ СУЩЕСТВУЕТ
- **97.7% infeasibility rate**

Это значит, что Z80-компилятор ОБЯЗАН декомпозировать почти каждую нетривиальную
функцию на "острова" (islands), которые помещаются в регистровый файл.

Island decomposition — это не оптимизация. Это **математическая необходимость**.

### Treewidth: теория врёт

Интересная деталь: 99.5% случайных графов интерференции имеют treewidth <= 3
(классически решаемы). Но компиляторные графы — плотнее: 53.7% реальных функций
имеют treewidth >= 4. Случайная модель не описывает реальные программы.

### Five-Level Pipeline

Ни один метод не справляется со всем:

1. **Table lookup** (83.6M записей, O(1)) — ≤6 переменных
2. **Graph decomposition** через cut vertices — O(1) на компоненту
3. **GPU brute-force** — ≤12 переменных, секунды
4. **CPU backtracking** (1000-4000× pruning) — ≤15 переменных, <1 секунды
5. **Island decomposition + Z3** — >15 переменных

Каждая функция попадает на один из этих уровней. Pipeline — полный.

## Глава 5: SBC A,A — универсальный conditional

Посреди ночи 26 марта, между второй и третьей чашкой чая, GPU начал выдавать
решения для branchless idioms. И в каждом втором появлялась одна и та же
инструкция:

```asm
SBC A,A       ; A = 0x00 если carry=0, A = 0xFF если carry=1
```

Это carry-to-mask trick. Одна инструкция превращает один бит (carry flag) в
полный байт: 0x00 или 0xFF. В комбинации с RLCA (которая кладёт знаковый бит
в carry), это даёт мгновенное branchless определение знака.

Вот зоопарк идиом, где SBC A,A играет главную роль:

```asm
; bool(A): 0 → 0, nonzero → 1
LD B,A        ; save
NEG           ; sets carry if A != 0
ADC A,B       ; A = 0+0+carry (carry propagated through NEG)
              ; 3 инструкции, 16T

; NOT(A): 0 → 1, nonzero → 0
NEG           ; carry if nonzero
SBC A,A       ; A = 0xFF if nonzero, 0x00 if zero
INC A         ; 0xFF → 0, 0x00 → 1
              ; 3 инструкции, 16T

; ABS(A): branchless absolute value
LD B,A        ; save original
RLCA          ; sign bit → carry
SBC A,A       ; 0xFF mask if negative, 0x00 if positive
XOR B         ; conditional complement
SBC A,B       ; adjust
ADC A,B       ; fix carry
              ; 6 инструкций, 24T — без единого branch!

; sign-extend A → HL (8-bit → 16-bit)
ADC A,L       ; overflow if A >= 128
SBC A,A       ; 0xFF mask for negative
LD H,A        ; H = sign extension byte
              ; 3 инструкции, 12T
```

SBC A,A — это Z80-эквивалент x86 CMOV. Одна инструкция заменяет условный
переход. Каждый branchless idiom в таблице — от bool(A) до ABS(A) — использует
именно эту инструкцию как foundation.

Ещё одна находка: `half(A) = RRA`, не `SRL A`. Обе делят на 2, но RRA стоит
4T (без CB-prefix), а SRL A — 8T (с CB-prefix). Разница — в encoding:
RRA — однобайтовая, SRL A — двухбайтовая. GPU выбирает RRA каждый раз.

И `CPL` — инструкция, которую мы *забыли* добавить в начальный pool. Bitwise
NOT за 1 инструкцию (4T). Когда добавили — она немедленно появилась в
оптимальных решениях. Pool design matters.

## Глава 6: День второй — Floating Point и BCD

27 марта. Утро. Торт съеден. GPU работали всю ночь. Пора заняться вещами,
которые обычно считаются невозможными на 8-битном процессоре.

### Z80-FP16: формат, который любит Z80

Стандартный IEEE 754 FP16 хранит биты как [SEEEEEEMMMMMMMMM] — знак, экспонента
и мантисса перемешаны внутри двух байт. На Z80, где нет barrel shifter, извлечь
экспоненту из IEEE формата стоит ~30-40T.

Наш формат: **H=[EEEEEEEE] L=[SMMMMMMM]**

Экспонента — целый байт H. Мантисса со знаком — байт L.

Почему это гениально:

```asm
; ×2 (удвоение float):
INC H         ; 4T. Всё. Экспонента + 1 = умножение на 2.

; ÷2 (деление float пополам):
DEC H         ; 4T.

; ABS:
RES 7,L       ; 8T. Убрать знаковый бит.

; NEG:
SET 7,L       ; 8T. Или: RES 7,L для положительного.
```

IEEE FP16 потратил бы 30-40T на ×2. Наш формат — 4T. Потому что byte-aligned
exponent — это не компромисс, это **оптимизация под архитектуру**.

Полный пакет `pkg/fp16/` — Go library с конвертацией IEEE ↔ Z80-FP16, арифметикой
и проверками.

### BCD: когда DAA наконец заработал

Binary-Coded Decimal — каждый nibble хранит десятичную цифру (0-9). Z80 имеет
инструкцию DAA (Decimal Adjust Accumulator), но её поведение зависит от H-flag
(half-carry), который не все эмуляторы реализуют правильно.

Мы исправили H-flag в нашем исполнителе. И DAA ожил:

```asm
; BCD ×2 (удвоить BCD число):
ADD A,A       ; binary double
DAA           ; decimal adjust: 0x12 → 0x24, 0x49 → 0x98
              ; 2 инструкции. GPU доказал корректность для всех 100 BCD-входов.

; 100's complement (BCD нegate):
NEG           ; binary negate
DAA           ; decimal adjust
              ; 2 инструкции. Красиво.
```

BCD ×2 = `ADD A,A; DAA` — две инструкции, GPU-proven. Это одна из тех вещей,
которые опытные Z80-программисты "знали", но никто не доказывал формально.
Теперь доказано: для всех 100 допустимых BCD-входов (0x00-0x99), результат
корректен.

## Глава 7: 21 инструкция на всю арифметику

Вечером 27 марта мы собрали все таблицы вместе и посмотрели на объединение:

```
mul8:  14 инструкций   (A × K → A)
mul16:  5 инструкций   (A × K → HL)
div8:   6 инструкций   (A / K → A)
───────────────────────
Union: 21 уникальная инструкция
```

**21 инструкция из ~700 в Z80 ISA.** Это 2.7%.

Cross-table analysis показывает удивительную структуру:

| Пересечение      | Инструкции | Количество |
|-------------------|-----------|----------:|
| mul8 only         | ADD A,A, ADD A,B, LD B,A, RLA, SBC A,B, RLCA, NEG, RRCA, ADC A,B, SUB B, RRA, SBC A,A, ADC A,A | 13 |
| mul16 only        | SUB HL,BC, SWAP_HL | 2 |
| div8 only         | LD A,H, SHR_HL | 2 |
| mul8 ∩ div8       | SRL A | 1 |
| mul16 ∩ div8      | ADD HL,BC, ADD HL,HL, LD C,A | 3 |
| **mul8 ∩ mul16**  | *(empty)* | **0** |

**mul8 и mul16 не делят ни одной инструкции.** Zero overlap.

Это архитектурно логично: mul8 живёт в мире A-регистра (8-битные сдвиги и
сложения), mul16 — в мире HL (16-битные). Деление — мост между мирами:
reciprocal multiplication в HL, затем извлечение в A.

Три Top-3 инструкции по частоте:

```
mul8:  ADD A,A (39.3%) + ADD A,B (22.8%) + LD B,A (12.3%) = 74.4%
mul16: ADD HL,HL (54.0%) + ADD HL,BC (30.0%) + LD C,A (10.4%) = 94.4%
```

Три инструкции — три четверти всех решений. Удвоение + прибавление + сохранение.
Shift-and-add. Самый древний алгоритм умножения. GPU переоткрыл то, что
математики знали тысячелетия — и доказал, что ничего лучше нет.

### Universal 21-op kernel

На основе этих 21 инструкций мы построили CUDA-ядро `z80_universal21.cu`:
depth-10 search за 2.3 часа на одном GPU. Единый kernel, который может искать
оптимальные последовательности для ЛЮБОЙ арифметической функции.

## Глава 8: Approximate Superoptimization

К полуночи 27 марта пришла идея: а что если target — не точная функция?

Multi-target search: вместо одной функции ищем последовательность, которая
минимизирует ошибку для нескольких функций одновременно. Или — ищем точный match
для приближённой функции.

### LUT generators

Главная находка: 12 байт Z80-кода могут *сгенерировать* 256-байтную lookup
table. Вместо хранения таблицы в ROM — генерация при загрузке. Ratio 20:1.

```
gray_encode(A) = A XOR (A >> 1)
```

GPU нашёл точное решение:

```asm
; gray_encode: EXACT at 3 ops
LD B,A        ; save
SRL A         ; A >> 1
XOR B         ; A XOR original
              ; 3 инструкции, 16T. Точно для всех 256 входов.
```

Другие результаты multi-target search:

| Target         | Ops | Accuracy | Notes                        |
|----------------|----:|----------|------------------------------|
| gray_encode    |   3 | EXACT    | A XOR (A >> 1)               |
| is_pow2        |   7 | EXACT    | boolean: true if power of 2  |
| sqrt f3.5      |  ~8 | 92%      | fixed-point, ±10             |
| sqrt f4.4      |  ~7 | 96%      | better scaling, faster convergence |

**Algorithmic compression**: 12 байт кода вместо 256 байт ROM. Для ZX Spectrum
с 48KB RAM это не мелочь — это 0.5% памяти сэкономлено на одну таблицу.

Scaling matters: f4.4 (4 бита целая, 4 дробная) сходится быстрее, чем f3.5.
Больше дробных бит — точнее, но труднее для поиска. Но формат f4.4 "попадает
в sweet spot" для 8-битной арифметики.

### Trimmed multi-target kernel

`z80_multitarget_13.cu` — 13-op pool, 15 функций одновременно. Один запуск
GPU тестирует каждую последовательность против всех 15 targets. Если хоть одна
совпадает — записываем.

## Глава 9: Nanz на GPU — от безумной идеи до 4/4 за одну ночь

Около десяти вечера 27 марта в ddll (наш inter-session messenger) приходит
сообщение от коллеги из minz-vir:

> *"А что если Nanz скомпилировать в GPU shaders?"*

Nanz — это наш язык. Компилируется в VIR (Virtual Intermediate Representation),
потом в Z80-ассемблер. Вопрос — а можно ли вместо Z80 выдать CUDA? Metal?
OpenCL? Vulkan?

Ответ пришёл за одну ночь.

### Ночь: CUDA

Первый proof of concept: функция `double(x)` на Nanz.

```nanz
fun double(x: u8) -> u8 {
    return x + x
}
```

Компилятор генерирует:

```cuda
__device__ uint8_t nanz_double(uint8_t x) {
    uint8_t r = (uint16_t)x + (uint16_t)x;
    return r & 0xFF;  // u8 semantics: mask at codegen.go:192
}
```

Запуск на RTX 4060 Ti: **256/256 correct.** Каждый возможный 8-битный вход
дал тот же результат, что Z80-эмулятор.

Ключевая деталь — `& 0xFF` на строке 192 в `codegen.go`. Без этой маски GPU
считает в 32-bit int и переполнение даёт другой результат. С маской — u8
семантика сохранена. Один символ, пять байт кода, разница между работающим
компилятором и broken.

### Ночь: OpenCL

Та же функция, OpenCL backend:

```opencl
__kernel void nanz_test(__global uchar* in, __global uchar* out) {
    uint idx = get_global_id(0);
    uchar x = in[idx];
    uchar r = (ushort)x + (ushort)x;
    out[idx] = r;
}
```

Запуск на NVIDIA (через OpenCL ICD): **256/256 correct.**

### Ночь: Vulkan на AMD

SSH на i3 (macaron), где стоит Radeon RX 580. Vulkan compute shader:

```glsl
#version 450
layout(local_size_x = 256) in;
layout(binding = 0) buffer Input  { uint data_in[];  };
layout(binding = 1) buffer Output { uint data_out[]; };

void main() {
    uint idx = gl_GlobalInvocationID.x;
    uint x = data_in[idx] & 0xFF;
    uint r = (x + x) & 0xFF;
    data_out[idx] = r;
}
```

Dispatch. Ждём. Результат: **256/256 correct.**

Vulkan на AMD RX 580 через Mesa RADV driver. Без ROCm. Без проприетарных
драйверов. Open-source stack от кернела до результата.

### Утро: Metal на Apple M2

MacBook Air. Последний backend. Metal compute shader:

```metal
kernel void nanz_test(device uint8_t* in  [[buffer(0)]],
                      device uint8_t* out [[buffer(1)]],
                      uint idx [[thread_position_in_grid]]) {
    uint8_t x = in[idx];
    uint8_t r = (uint16_t)x + (uint16_t)x;
    out[idx] = r;
}
```

**256/256 correct.**

### Утро: Auto-detect for loops

Последний штрих: компилятор распознаёт `for` цикл с независимыми итерациями
и автоматически превращает его в GPU dispatch. Каждая итерация — один thread.

```nanz
for i in 0..256 {
    result[i] = double(i)
}
```

→ GPU kernel с 256 потоками. **256/256 correct.**

**mir2gpu merged to master.**

Заголовок этой ночи:

> *"One language, four APIs, three vendors, fifty years of hardware, zero bugs."*

Nanz — язык, спроектированный для 8-битных процессоров 1976 года — работает
на GPU 2024 года. И даёт ровно тот же результат. Потому что u8 — это u8,
неважно на каком кремнии.

## Глава 11: День третий — Enrichment и архитектура

*28 марта 2026, утро до поздней ночи*

День начался с кофе и вопроса: что делать с 37.6 миллионами таблиц?

### Gray Decode: Exact за секунду

RX 580 через Vulkan. Пять операций в пуле вместо тринадцати. Глубина 13.
Результат: EXACT. Менее одной секунды. Три NVIDIA на depth-12 с тринадцатью
операциями не нашли за три часа. А AMD на пяти операциях — мгновенно.

Урок: меньше свободы → глубже поиск → лучше результат. Парадокс ограничений.

### Z flag — write-only

Ни одна инструкция Z80 не читает Z flag как вход для вычислений.
Доказательство: перебор всех 26 релевантных инструкций + индукция.
Z→CY невозможно без branch. Точка.

Для компилятора это значит: CY — единственный branchless bool.
SBC A,A превращает CY в маску за 4 такта. Из маски — MIN, MAX, ABS, CMOV.
Библиотека branchless примитивов, каждый verified на 65536 входах.

### div3 = A×171>>9

Перебор всех множителей 1-511 для делителей 3,5,7,10.
Только деление на 3 имеет точный множитель: 171. Ноль ошибок на всех 256 входах.
Никакая публикация не упоминает это для Z80.

### Enrichment: 43% скрытой инфебильности

Таблицы говорили "37.6 миллионов раскладок feasible". Но когда мы добавили
стоимости операций — 43% из них не имеют аккумулятора A. Для ALU-кода
они фактически бесполезны без дополнительных перемещений.

9% — идеальны. Девять процентов из миллионов.

CALL overhead: наивный PUSH/POP = 34 такта. Умный (свободный регистр → IX → стек) = 17.
Пятьдесят процентов экономии на каждый вызов функции.

### Corpus: 820 функций, ноль умножений

VIR прислал corpus из реального компилятора. 820 функций.
move = 34%. Каждая третья инструкция — перемещение между регистрами.
Именно это оптимизирует register allocator.

mul = 0%. Ноль. На Z80 никто не умножает.

246 уникальных сигнатур. Мы предсказывали "менее 500". Попали.

### O(1) Regalloc

Сигнатура: (форма графа, мешок операций) → хеш → таблица → ответ.
Порядок операций не важен для стоимости. Interference graph уже кодирует порядок жизни.

91% функций — O(1) lookup. 8% — GPU партиционирование. 1% — Z3.
Ноль процентов нерешаемых.

### 19 и 20 переменных за ночь

GPU0: 19 переменных, 275 миллиардов вариантов разбиения. Семь минут.
GPU1: 20 переменных, 1.1 триллиона. Тридцать минут.
i5: второй вариант 20v параллельно.

Весь corpus — 172 функции с 7+ переменными — разбит оптимально. 100%.

### RMDA и рисование через CALL

Максим Мучкаев из Самары. 256-байтные интро для Spectrum.
CALL пушит адрес возврата на стек. Если стек = экранная память,
CALL рисует два пикселя. 8.5 тактов на байт — быстрее любого LD.

А Introspec добавляет: POP HL; LDD; LD A,(HL); LD (BC),A — 
чанки которые одновременно адреса и данные. 16 значений = 16-цветная палитра.

Два техника можно совместить. CALL генерирует чанки, decode рисует.
GPU ищет оптимальную таблицу конверсии.
