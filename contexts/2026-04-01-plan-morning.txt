● Here's the full feasibility breakdown across all 5 questions:

  ---
  18-Register Z80 Model: Feasibility Analysis

  Q1 — IXH/IXL as ALU src: new tables needed?

  Feasibility: HIGH for adding to regalloc tables, LOW ROI for mul8/div8
  sequences specifically.

  From z80_register_graph.json (already modeled): ADD/SUB/AND/OR/XOR/CP/INC/DEC
  all accept IXH/IXL as src at 8T. ADC/SBC do not (no ED+prefix combo exists).
  The H/L↔IXH/IXL conflict (16T via EX DE,HL trick) must be encoded as a
  move-cost penalty, not an interference edge — they don't conflict
  simultaneously, they're just expensive to transfer between.

  For mul8 via IXH as temp: current mul8 tables use B as temp (4T save+restore),
   IXH is 16T (LD IXH,A + LD A,IXH). Only useful when all 7 GPR are occupied,
  which is rare. The existing sequences are already optimal for typical
  patterns.

  For regalloc tables: currently locSets8 has 4 options (locs 0-6). Adding:
  - {IXH,IXL,IYH,IYL} — must be IX/IY half
  - {0..6,10..13} — any 8-bit including IX halves

  → 6 locSets8 × 3 locSets16. For nv=6: (6/4)^6 ≈ 3.4× more combos → ~280M
  total, ~100M feasible. GPU time: ~15-25 hours (weekend run). Doable.

  ---
  Q2 — EXX inside computation path

  Feasibility: HIGH for specific idioms, LOW for general scheduling.

  The critical constraint: EXX is a bank flip, not a register move. You can't
  EXX to access just B' — all 6 pairs swap simultaneously. This rules out
  treating shadow regs as individually-addressable locations in the interference
   graph model.

  What is tractable:
  - HLH'L' 32-bit add (see Q3) — a single well-defined idiom
  - Dual-B add: ADD A,B; EXX; ADD A,B; EXX — adds both the main and shadow B to
  A (12T total, useful for histogram-like accumulation across two independent
  counters)
  - EXX as bulk save: preserve B,C,D,E,H,L to shadow at 4T (vs 66T PUSH×6) —
  perfect for interrupt handlers or short leaf functions that need all GPR

  For GPU search with EXX in the path: state space doubles (need to track which
  bank is active). Feasible as a specialized kernel for a fixed pattern class
  (e.g., "find all 4-instruction sequences using exactly one EXX pair"), but not
   as a drop-in into the current 3-stage pipeline.

  Near-term: encode the useful idioms as named patterns in data/u32_ops.json +
  data/arith16_new.json.

  ---
  Q3 — HLH'L' 32-bit: overhead vs DEHL

  Cost breakdown:

  HLH'L' add (two independent 32-bit operands):
    ADD HL, BC   ; 11T  low: HL += BC
    EXX          ;  4T  flip to shadow
    ADC HL, BC   ; 15T  high: H'L' += B'C' + carry  ← B' and C' are now current
  HL bank!
    EXX          ;  4T  restore
    = 34T

  DEHL add (general, operand in BCXX):
    ADD HL, low   ; 11T  low half
    EX DE, HL     ;  4T  DE→HL (to use ADC HL,rr)
    ADC HL, high  ; 15T  high + carry
    EX DE, HL     ;  4T  restore
    = 34T  — same! But requires high addend already in a reg pair

  Where HLH'L' wins: when both the low and high halves of the addend are in the
  same register pair (BC in main + B'C' in shadow). The DEHL model requires the
  high addend to be in a separate pair from the low addend — in practice this
  forces a spill when all pairs are busy, pushing to 54T.

  HLH'L' structural advantage: uses only HL in the visible main bank. DE stays
  free for pointer arithmetic. This is significant for compiler use — DEHL
  "hogs" both DE and HL, conflicting with pointer-heavy code.

  Recommendation: add HLH_L_add, HLH_L_shl1, HLH_L_neg to data/u32_ops.json.
  Mark as preferred when DE is needed for other purposes.

  ---
  Q4 — GPU regalloc tables: 11 locs feasible?

  IXH/IXL/IYH/IYL (11 locs total): YES, feasible.

  ┌────────────────────────┬─────────┬─────────────┬──────────────┬──────────┐
  │         Config         │ locSets │ nv=6 combos │ Est. entries │ GPU time │
  ├────────────────────────┼─────────┼─────────────┼──────────────┼──────────┤
  │ Current (locs 0-9)     │ 4+3     │ 13M         │ 83.6M        │ ~6h      │
  ├────────────────────────┼─────────┼─────────────┼──────────────┼──────────┤
  │ +IX halves (locs 0-13) │ 6+3     │ 44M         │ ~280M        │ ~20h     │
  └────────────────────────┴─────────┴─────────────┴──────────────┴──────────┘

  ~20 hours on current 2×RTX 4060 Ti hardware. Schedule as overnight run.

  Shadow registers (18 locs): NOT feasible as a flat table.

  Adding B'..L' as 6 more locSets: (12/4)^6 = 729× more combos → ~61 billion
  entries. At ~37M feasible/6h that's ~350 GPU-days. Hard no.

  Alternative for shadow: a separate EXX-zone table. Inside an EXX zone the
  available locations are exactly {B'=loc0, C'=loc1, ..., L'=loc5} — a
  6-register subproblem with no new loc variety. The zone table has the same 
  size as the current main-bank table (same structural combos). Compose at zone
  boundary with a 4T EXX overhead. This is elegant: zone allocation = existing
  pipeline, boundary cost = constant.

  ---
  Q5 — Datalog / --lir to VIR integration

  Feasibility: HIGH for fast path, MEDIUM for full reimagination.

  Our enriched tables are already a materialized Datalog database — (shape,
  op_bag) → (cost, assignment) is exactly a Datalog EDB. The pkg/regalloc API
  (IndexOf + Lookup) is the query interface.

  The --lir backend's Datalog facts (interference graph, def-use chains, loc
  constraints) map directly to the Shape type in pkg/regalloc/index.go. The
  missing bridge is the VIR → Shape translation.

  Three integration paths:

  Path A — Fast (1-2 weeks, HIGH feasibility):
  In minz-vir: add a liveness pass that emits pkg/regalloc.Shape structs → call
  IndexOf → table lookup. Covers ~79% of corpus without GPU. Pure Go, no new
  dependencies. Essentially just plumbing — the Shape type already has
  everything.

  Path B — Datalog rules from --lir (2-3 months, MEDIUM feasibility):
  Extract the constraint rules from --lir backend, port to Souffle syntax,
  compile against our enriched table as EDB. Covers the 21% overflow (currently
  hitting Z3/GPU). Requires Go→Souffle bridge via subprocess or WASM.

  Path C — Unified reimagination (quarter+, MEDIUM feasibility):
  VIR lowering as Datalog query rules, enriched tables as EDB, Z3 as escape
  hatch. Single declarative language for the full pipeline. High reward but
  requires committing to Souffle as a build dependency.

  Immediate next step: Path A is a straightforward PR to minz-vir — the
  interference graph is already constructed somewhere in the VIR backend, it
  just needs to be wired to the lookup.

● Yes! Both are zone-invariant (untouched by EXX and EX AF,AF') and have
  ED-prefix load/store:

  I register — 8-bit spill, 18T round trip:
  LD I, A    ; 9T  ED 47 — save A to I
  LD A, I    ; 9T  ED 57 — restore (clobbers S,Z,H,N flags; P/V = IFF2 state!)

  byte of the vector table address → using I as spill requires IM 0 or IM 1 (ZX

❯ oh, cool. R is a 1 bit. materialize flag to A as 00/FF (cheaper than 0/1) and 
 do LD R,A for 1 bit storage                                                    

● Exactly — and the round-trip cost works out cleanly with the canonical
  0x00/0xFF representation:

  Save (A already has 0x00/0xFF from SBC A,A or similar):
  LD R, A        ; 9T  bit7=0→false, bit7=1→true. lower bits don't matter.

  Restore (lower 7 bits are garbage from refresh counter, bit 7 is intact):
  LD A, R        ; 9T  bit7 correct, bits 0-6 = whatever refresh count
  ADD A, A       ; 4T  bit7 → CY (shift left, discards lower garbage)
  SBC A, A       ; 4T  materialize CY → 0xFF or 0x00

  Total: 9T save, 17T restore = 26T round trip for a boolean flag that crosses
  any zone boundary.

  Compare to IX half for booleans: 8T + 8T = 16T round trip, but uses one of 4
  precious IX slots.
  R register: 26T but uses no named register slot — R is otherwise just the DRAM
   refresh counter doing its own thing.

  And the save is even cheaper if the flag is already in CY (before
  materializing to A):
  SBC A, A       ; 4T  CY → 0x00/0xFF
  LD R, A        ; 9T
  = 13T save from raw CY. Full round trip from CY → store → restore → 0x00/0xFF
  = 13T + 17T = 30T.

  The recovery pattern ADD A,A; SBC A,A is the exact reverse of SBC A,A — nice
  symmetry. You push the bit into CY on the way out the same way you pulled it
  from CY on the way in.

  Updated cost table entry for R:

  ▎ R register: 9T save (from 0x00/0xFF) + 17T restore = 26T. Capacity: 1 
  boolean. Recovery: LD A,R; ADD A,A; SBC A,A → 0x00/0xFF. Costs no named 
  register slot. IM 0/1 only; flags clobbered on restore.
